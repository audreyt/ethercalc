# Packaging EtherCalc for Sandstorm

This branch is the Sandstorm-distribution source tree for EtherCalc.
It's the `chunked-upload` TypeScript rewrite plus four pieces of
Sandstorm-specific packaging glue:

  - `sandstorm-pkgdef.capnp` ‚Äî the Sandstorm package definition
    (Cap'n Proto), signed with the EtherCalc app ID
    `a0n6hwm32zjsrzes8gnjg734dh6jwt7x83xdgytspe761pe2asw0`.
  - `pgp-signature`, `pgp-keyring` ‚Äî PGP material used by `spk`
    to sign/verify the package.
  - `run_grain.sh` ‚Äî the grain's boot script. Starts Miniflare on
    port 33411 (the port sandstorm-http-bridge proxies), waits for
    `/_health`, then migrates any legacy `/var/dump*` content into
    the Worker's Durable Object storage on first boot.
  - `sandstorm-files.list` ‚Äî stub; regenerated by `spk dev`.

## Building the .spk

On a machine with Sandstorm + `spk` installed:

    # 1. Ensure bun is available ‚Äî run_grain.sh assumes `/usr/local/bin/bun`.
    curl -fsSL https://bun.sh/install | bash

    # 2. Install workspace deps (one-time).
    bun install

    # 3. Build the client bundles + curated assets/ dir.
    bun run build:assets

    # 4. Generate sandstorm-files.list by tracing a dev-mode run.
    spk dev
    # Create a grain, poke around, stop; spk will write the file list.

    # 5. Build the signed package.
    spk pack ethercalc.spk

The resulting `ethercalc.spk` is the upgrade artifact for Sandstorm's
app market listing.

## Legacy-grain migration path

Before this release, Sandstorm's EtherCalc ran the LiveScript codebase
with `OPENSHIFT_DATA_DIR=/var` and no Redis. Per-grain state lived in:

  - `/var/dump.json` ‚Äî single-blob JSON, older layout.
  - `/var/dump/snapshot-<room>.txt` + `/var/dump/audit-<room>.txt` ‚Äî
    directory layout, with `\\n`/`\\r`/`\\\\`-escaped audit lines.

On the first boot after upgrade, `run_grain.sh` detects either layout
and pipes it through `ethercalc migrate --source file:///var‚Ä¶ --token
sandstorm-grain-local`. This is idempotent ‚Äî re-running would just
overwrite each room with the same value ‚Äî but we drop a
`/var/.migrated` sentinel to avoid paying the per-boot probe cost.

What carries over:

  - Directory layout ‚Üí snapshots + audit logs (legacy keeps only these
    on disk; log/chat/ecell are in-memory only and lost across any
    Redis-unavailable restart).
  - JSON blob layout ‚Üí everything (snapshots, logs, audit, chat, ecell,
    timestamps).

What does not carry over (either layout):

  - Cron triggers (`cron-list`, `cron-nextTriggerTime` ‚Äî in-memory only
    in legacy).
  - socket.io session IDs (transport detail, not user data).

If `run_grain.sh` reports a migration failure, it does NOT drop the
sentinel, so the next grain boot retries. Check the grain log.

## Known packaging gap: worker runtime in the grain sandbox

A live-server test (native Ubuntu 24.04 x86_64, `spk dev` over the
installed legacy v2017.02.21.0) confirmed the migration-trigger path
end-to-end:

  1. Legacy grain runs, writes `/var/dump.json` with user content.
  2. After the `spk dev` swap the new `run_grain.sh` fires, finds
     `/var/dump.json`, and invokes `ethercalc migrate --source
     file:///var/dump.json --target http://127.0.0.1:33411 --token
     sandstorm-grain-local` ‚Äî all args correct, planted content
     intact.
  3. Migrate blocks on `/_health` waiting for the worker to come up.

What's *not* verified yet is the worker itself booting inside the
grain sandbox. `wrangler dev` + Miniflare reach out to Cloudflare
metadata at startup (the `setupCf` call), but Sandstorm grains have
**no outbound network by default**. So `wrangler dev` panics with
`Unexpected server response: 101` before binding 33411, which causes
`ethercalc migrate`'s `waitForHealth` loop to time out.

The migration **logic** is fine (run_grain.sh invoked it with the
right args); the gap is the worker runtime expecting dev-time CF
network access that the grain doesn't grant.

**Recommended fix** (separate packaging work): bundle `workerd`
standalone via `wrangler deploy --dry-run --outdir=dist/` at
`spk pack` time, then launch `workerd serve dist/config.capnp` in
`run_grain.sh` instead of `bun bin/ethercalc`. `workerd` doesn't need
CF metadata to start ‚Äî it just needs the pre-bundled worker and the
DO/KV/R2 configuration. Also shrinks the package significantly (no
bun, no `node_modules`).

Interim workaround: add `ipNetwork` to the pkgdef's `bridgeConfig` so
the user can grant outbound HTTP on first grain start. Less ergonomic
(extra permission prompt) but unblocks `wrangler dev` without
repacking.

### Grain sandbox env vars (applied in run_grain.sh)

Three env-var workarounds are needed for Bun/Wrangler to get past the
sandbox's defaults; they're set automatically by `run_grain.sh`:

  - `HOME=/var/home`       ‚Äî Wrangler's `os.homedir()` ENOENTs without this
  - `TMPDIR=/var/tmp`      ‚Äî Bun's `createFakeTemporaryNodeExecutable` wants
                              a writable temp dir; the grain's /tmp fights it
  - `BUN_INSTALL_CACHE_DIR=/var/bun-cache` ‚Äî keep bun's cache grain-local

In addition, the pkgdef must run with `spk dev --proc` so `/proc` is
mounted inside the sandbox ‚Äî Bun's runtime relies on `/proc/self/exe`
to bootstrap. Packaged grains get this via Sandstorm's standard proc
mount; only dev mode needs the explicit flag.

## Grain-local migrate token

`run_grain.sh` hard-codes `ETHERCALC_MIGRATE_TOKEN=sandstorm-grain-local`.
The token gates `PUT /_migrate/seed/:room` on the worker. Because the
worker is only reachable on localhost from within the grain sandbox,
a fixed token is safe ‚Äî grain isolation is the primary defense, the
token is belt-and-suspenders.

The CLI shim in `bin/ethercalc` forwards this env var into wrangler
via `--var ETHERCALC_MIGRATE_TOKEN:<value>` so the Miniflare worker
sees the same value as the migrate command.
